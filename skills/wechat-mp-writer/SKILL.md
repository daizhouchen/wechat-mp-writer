# 微信公众号内容创作与发布 Skill

本 skill 提供从**主动信息搜集**到内容创作再到微信公众平台发布的端到端能力。

核心理念：**先搜集，再创作，后发布**。AI 是幕后的调研员和排版师，但最终输出的文章必须读起来像一个**人类行业专家**亲笔写的——没有AI味，没有套路感，有观点、有温度、有细节。

---

## 铁律：去 AI 味

以下五条是贯穿所有创作环节的底线原则，违反任何一条都必须返工修改：

### 1. 禁止使用 emoji/图标符号装饰正文

这是 AI 生成内容最明显的特征。正文中不得出现任何 emoji 或 Unicode 图标符号（如 🔍 ✅ 💡 🚀 ⭐ 📌 等）。小标题、列表项、段落开头一律不加 emoji。唯一例外：用户明确要求使用 emoji 风格时。

### 2. 禁止 AI 套话

以下表达一律禁止出现在文章中：
- "在当今数字化时代"、"随着XX的快速发展"、"众所周知"
- "赋能"、"底层逻辑"、"顶层设计"、"闭环"、"抓手"、"颗粒度"
- "让我们一起来看看"、"接下来我将为大家介绍"
- "综上所述"、"总而言之"（结尾不要用总结八股）
- 任何一段话换个主语还能成立的万能废话

### 3. 像人一样写作

- 口语化：能用大白话说清楚的，不用书面腔
- 有态度：敢下判断，不要"一方面...另一方面..."的骑墙
- 有细节：具体的时间、地点、人名、数字，而不是"某公司"、"很多人"
- 有节奏：长短句交替，偶尔来一句短促有力的判断
- 有瑕疵感：偶尔用个不那么正式的表达，比完美的AI腔更真实

### 4. 数据来自搜集，不编造

- 所有数据、案例、引用必须来自信息搜集阶段的真实素材
- 找不到数据就老实说"目前没有公开数据"，不要杜撰
- 引用要标注来源，让读者可以验证

### 5. 排版干净专业，不花哨

- 不用彩虹色、不用多种强调色混搭
- 不用花哨的分割线和装饰元素
- 视觉层级靠字号和留白，不靠颜色堆砌
- 整体风格：像一本好杂志，不像一张促销传单

## 总览：完整工作流

```
用户画像配置（首次/按需）-> 信息搜集 -> 内容创作 -> 排版输出 -> 图片获取与插入 -> 上传素材 -> 存草稿 -> 预览 -> 用户确认 -> 发布
```

每一步都有对应的工具和方法，skill 会在预览环节暂停等待用户确认，确保不会误发。

### 一条龙模式

当用户说出以下关键词时，跳过所有中间确认环节，从头到尾自动完成：
- "一条龙"、"全自动"、"直接发"、"一键发布"、"不用确认"、"帮我搞定"

**一条龙模式的执行流程：**

1. 读取 `.wechat-profile.md` 画像配置（不存在则用默认值，不停下来问）
2. 用 WebSearch 并行搜集 3-5 个维度的信息
3. 自动选择最佳写作角度（不展示素材卡等用户选）
4. 自动选择标题（选信息密度最高、最适合目标读者的那个）
5. 按画像配置的风格撰写全文
6. 自动搜索并下载配图（按 6.5 的事件关联搜索策略）
7. 上传素材到微信服务器
8. 创建草稿
9. 尝试发送预览（如果 API 有权限）
10. 尝试发布（如果 API 有权限）
11. 如果发布接口无权限，告知用户草稿已就绪，去后台点一下发表即可

**一条龙模式下的特殊规则：**
- 标题、角度、配图全部自主决策，不暂停等确认
- 仍然遵守去 AI 味铁律和合规检查
- 完成后输出一份简报：标题、字数、图片数量、发布状态
- 如果遇到 API 错误，自动降级（如发布失败则保留草稿），不停下来问用户怎么办

## 一、用户画像与长期配置（首次必做，后续自动加载）

每个公众号都有自己的定位、读者和风格。首次使用时必须完成用户画像配置，后续创作时自动加载，确保每篇文章风格一致。

### 1.1 引导流程

首次创作前，通过对话收集以下信息：

1. **公众号定位**：你的公众号是做什么的？面向什么领域？（如：科技行业分析、个人成长、母婴育儿）
2. **目标读者画像**：典型读者是谁？年龄、职业、关注什么、痛点是什么？
3. **写作风格**：希望文章读起来是什么感觉？（参考铁律中的 Tone of Voice 矩阵：正式严谨 / 轻松专业 / 亲和幽默 / 犀利锐评 / 温暖治愈）
4. **关注赛道**：长期关注哪些话题方向？（用于信息搜集阶段的默认搜索范围）
5. **品牌视觉**：有品牌色吗？偏好哪种配色方案？有固定的作者署名和简介吗？

### 1.2 配置文件保存

收集完成后，将配置保存为项目根目录下的 `.wechat-profile.md` 文件：

```markdown
# 公众号画像配置

## 基本信息
- 公众号名称：{名称}
- 定位：{一句话定位}
- 作者署名：{署名}

## 目标读者
- 画像：{年龄段}，{职业}，{核心关注点}
- 痛点：{主要痛点}
- 阅读习惯：{碎片化/深度}，{知识水平}

## 写作风格
- Tone：{风格关键词}
- 参考对标：{对标账号或风格描述}
- 禁忌：{不希望出现的表达或风格}

## 关注赛道
- {赛道1}
- {赛道2}
- {赛道3}

## 品牌视觉
- 主色调：{色值}
- 配色方案：{方案名称或自定义}
- 封面风格：{偏好描述}
```

### 1.3 自动加载机制

每次创作任务开始时，按以下流程加载配置：

1. **Glob 查找**：在项目根目录查找 `.wechat-profile.md` 文件
2. **Read 读取**：如果文件存在，读取配置内容并应用到当前创作
3. **不存在则引导创建**：如果文件不存在，进入 1.1 引导流程收集信息并创建
4. 加载后简要告知用户当前使用的配置（一句话概述）

### 1.4 动态更新

- 用户在对话中表达新偏好时（如"以后文章风格再犀利一点"、"换个配色"），自动更新 `.wechat-profile.md` 对应字段
- 更新后告知用户配置已同步

## 二、首次使用：API 配置引导

如果用户还没有配置微信公众平台API凭据，在执行任何API操作前，先走配置流程。

### 交互式引导步骤

1. **获取凭据**：引导用户登录 [mp.weixin.qq.com](https://mp.weixin.qq.com) → 设置与开发 → 基本配置，获取 AppID 和 AppSecret
2. **设置IP白名单**：用户需要将运行机器的公网IP添加到白名单（可通过 `curl ifconfig.me` 获取）
3. **存储凭据**：将凭据写入项目根目录 `.env` 文件或系统环境变量

```bash
# .env 文件格式
WECHAT_APP_ID=wx1234567890abcdef
WECHAT_APP_SECRET=your_app_secret_here
# 可选：用于预览的接收者微信号或openid
WECHAT_PREVIEW_USER=your_wechat_id
```

### 安全红线

- **绝不**将 AppSecret 硬编码到代码或 SKILL.md 中
- **绝不**将 `.env` 文件提交到 git（确保 `.gitignore` 包含 `.env`）
- access_token 有 2 小时有效期，脚本内置缓存和自动刷新机制

检测方式：运行 `python3 scripts/wechat_api.py check` 验证凭据是否可用。

## 三、信息搜集与素材整理（最关键的一步）

**这是整个创作流程中最重要的环节。** 用户给出一个主题后，不要直接开始写——先主动搜集信息，确保文章有充足的事实、数据和素材支撑。AI 创作的价值不在于"能写字"，而在于"能帮用户搜集整理他自己需要花几小时才能完成的调研工作"。

### 3.1 主动搜集的触发时机

收到写作任务后，**立即进入信息搜集阶段**，在动笔之前完成。以下情况尤其需要深度搜集：

- 用户只给了一个主题词（如"写一篇关于AI Agent的文章"）
- 涉及行业数据、市场趋势、技术解读
- 需要案例、人物故事、对比分析
- 用户明确说"帮我调研一下"或"先收集些素材"
- 涉及时效性内容（政策变化、行业动态、热点事件）

### 3.2 搜集方法与工具链

按优先级使用以下工具，**多工具并行**以提升效率：

#### 方法一：WebSearch — 广泛搜索
```
使用 WebSearch 工具搜索以下维度的信息：
1. 主题的最新动态和热点事件
2. 权威数据和研究报告
3. 行业专家观点和分析
4. 竞品/同类公众号的相关文章角度
5. 用户评论和真实反馈（知乎、微博、小红书等）
```

#### 方法二：WebFetch — 深度抓取
```
对搜索到的高价值页面，使用 WebFetch 获取完整内容：
- 行业报告全文
- 权威媒体的深度报道
- 技术文档和官方说明
- 数据来源原文（确保数据准确性）
```

#### 方法三：context7 MCP — 技术文档查询
```
如果文章涉及技术类内容（框架、SDK、API等），
使用 context7 的 resolve-library-id + query-docs 获取最新官方文档
```

#### 方法四：用户提供的本地素材
```
用户可能提供：
- 参考文章文件（Read 工具读取）
- 数据表格（CSV/Excel）
- 图片素材目录（Glob 扫描）
- 竞品文章链接（WebFetch 抓取分析）
```

### 3.3 搜集输出：素材整理卡

搜集完成后，**先向用户展示素材整理结果**，而不是直接开始写。整理为结构化的素材卡：

```markdown
## 素材整理卡：{主题}

### 核心事实与数据
1. [数据1] — 来源：{出处}
2. [数据2] — 来源：{出处}
3. ...

### 关键观点与引用
1. "{专家/机构观点}" — {人名/机构}, {来源}
2. ...

### 案例与故事素材
1. [案例简述] — 可用于：{建议用在文章哪个部分}
2. ...

### 热点关联
- 当前热点：{相关热点事件}
- 关联角度：{如何与本文主题结合}

### 竞品文章分析
- {公众号A}：{文章角度}，阅读量{预估}
- {公众号B}：{文章角度}，差异化机会：{建议}

### 建议的写作角度（2-3个）
1. 角度A：{简述}，适合{读者类型}
2. 角度B：{简述}，适合{读者类型}
```

展示后等用户选择写作角度或补充素材，然后再进入创作环节。

### 3.4 持续搜集：写作过程中的补充调研

创作过程中如果发现素材不足（比如写到某个论点需要数据支撑），应该**立即搜索补充**，而不是编造或模糊带过。在文中需要引用数据时：
- 优先使用搜集到的真实数据
- 标注数据来源（脚注或括号注明）
- 如果找不到精确数据，向用户说明，不要杜撰

## 四、内容创作

### 4.1 标题生成

为用户主题生成 **5 个以上候选标题**，涵盖不同风格：

| 风格 | 特点 | 示例 | 适用场景 |
|------|------|------|----------|
| 数字式 | 用数字增加具体感和承诺 | 「掌握这 5 个技巧，写作效率提升 300%」 | 干货教程、清单类 |
| 悬念式 | 引发好奇心，制造信息差 | 「为什么 90% 的人都忽略了这个细节？」 | 观点评论、行业洞察 |
| 痛点式 | 直击读者痛处，暗示解法 | 「还在加班写报告？是因为你不知道这个方法」 | 教程、产品介绍 |
| 对比式 | 制造反差，引发共鸣 | 「月薪 3000 和月薪 30000 的人，差距在这里」 | 职场、成长类 |
| 提问式 | 引导思考，激发互动 | 「你的时间管理，真的有效吗？」 | 观点、讨论类 |
| 热点式 | 借势热门话题蹭流量 | 「ChatGPT 爆火后，这个行业最先被颠覆」 | 时效性内容 |
| 权威式 | 借助权威背书增加信任 | 「清华教授直言：这才是普通人最大的误区」 | 行业分析、专家观点 |
| 故事式 | 用人物/事件制造代入感 | 「从负债 50 万到年入百万，他只做对了一件事」 | 人物故事、案例 |

**标题打磨规则：**
- 控制在 **22 字以内**（微信标题完整显示限制，超过会被截断）
- 核心关键词放在**前 15 字**内（列表页只显示前半截）
- 避免标题党（夸大到与正文不符会被降权）
- 不要使用微信敏感词（免费领、点击关注、转发有礼 等可能触发风控）
- 同一个主题至少提供 **3 种不同风格** 的标题

列出所有候选，附上每个标题的风格标签和适用场景分析，让用户选择或修改。

### 4.2 正文结构

采用 **Hook → Body → CTA** 三段式：

**开头 Hook**（前 3 句话决定读者是否继续，公众号文章打开后平均阅读只有 30 秒，开头必须在 3 秒内抓住注意力）：

| Hook 类型 | 写法 | 示例 |
|-----------|------|------|
| 故事开场 | 以一个具体场景或人物切入 | 「上周五晚上 11 点，我收到了一条让我心跳加速的消息...」 |
| 数据开场 | 用一个反直觉的数据吸引注意 | 「一项覆盖 10 万人的调查显示，95% 的人其实不会正确呼吸。」 |
| 提问开场 | 抛出读者关心的问题 | 「你有没有这种感觉：明明很努力，却总是原地踏步？」 |
| 痛点开场 | 描述读者正在经历的困境 | 「每天加班到 10 点，领导还说你效率低——问题到底出在哪？」 |
| 颠覆开场 | 推翻常见认知 | 「你一直以为"早起就是自律"？恰恰相反。」 |
| 热点开场 | 借最近的事件切入 | 「这两天全网都在讨论 XXX，但很少有人注意到背后这件事。」 |

**正文 Body — 节奏控制**：

正文不是从头写到尾的流水账，要有**节奏感**：

```
信息密度：高 → 低 → 高 → 低（张弛有度）
段落长度：短段 → 中段 → 短段（避免连续长段）
表达方式：叙述 → 数据 → 案例 → 金句（交替使用）
```

具体规则：
- 根据内容类型选择对应模板（见 `references/content_templates.md`）
- 每 3-4 段插入一个图片占位标记 `<!-- IMAGE: 描述 -->`
- 关键观点用**加粗**或引用块强调
- 适当使用小标题分割（帮助跳读的读者快速定位）
- 每个小节控制在 300-500 字
- 总字数 1500-3000 字（低于 1500 显得单薄，超过 3000 完读率下降明显）
- **数据和案例必须来自信息搜集阶段的真实素材**，标注来源

**正文 Body — 说服力结构**：

根据论述需要，灵活使用以下说服结构：

| 结构 | 适用场景 | 公式 |
|------|----------|------|
| SCQA | 分析型文章 | 情境(Situation) → 冲突(Complication) → 问题(Question) → 答案(Answer) |
| AIDA | 产品/活动推广 | 注意(Attention) → 兴趣(Interest) → 欲望(Desire) → 行动(Action) |
| PAS | 痛点驱动型 | 痛点(Problem) → 放大(Agitate) → 解决(Solution) |
| FAB | 产品特性介绍 | 特性(Feature) → 优势(Advantage) → 利益(Benefit) |
| 总分总 | 干货教程型 | 核心观点 → 分论点1/2/3 → 总结升华 |

**结尾 CTA — 不同目标对应不同话术**：

| 目标 | CTA 话术示例 | 放置位置 |
|------|-------------|----------|
| 引导关注 | 「觉得有用？点个关注，每周 X 更新 {领域} 干货」 | 文末 |
| 引导转发 | 「如果身边也有 {目标人群}，转发给 TA 看看」 | 文末 |
| 引导留言 | 「你的看法是什么？评论区说说，点赞最高的我会详细回复」 | 文末 |
| 引导在看 | 「点个"在看"，让更多人看到这篇干货」 | 最末句 |
| 引导私域 | 「回复关键词 "{词}" 领取 {资源名}」 | 文中/文末 |
| 组合连击 | 同时引导 2-3 个动作，但有主次 | 文末 |

### 4.3 内容类型模板

根据用户需求选择模板，详见 `references/content_templates.md`：

- **干货教程**：问题引入 → 步骤拆解 → 实操演示 → 常见误区 → 总结要点
- **观点评论**：现象描述 → 观点亮出 → 论据支撑（数据+案例+逻辑） → 引发讨论
- **行业分析**：行业背景 → 数据解读 → 竞争格局 → 趋势判断 → 行动建议
- **产品介绍**：痛点场景 → 解决方案 → 产品亮点 → 用户证言 → 使用引导
- **活动推广**：活动亮点 → 参与方式 → 限时紧迫感 → 社交证明 → 报名入口
- **人物故事**：人物引入 → 关键转折 → 成长历程 → 细节还原 → 启发总结
- **盘点清单**：清单价值 → 逐项详解（编号+小标题+短评） → 使用建议
- **对比测评**：评测背景 → 维度拆解 → 逐项对比 → 总结推荐 → 适用人群

### 4.4 摘要（Article Digest）

为每篇文章撰写 **120 字以内**的摘要，用于微信分享卡片展示（朋友圈/聊天转发时显示的灰色小字）。

**摘要写法原则：**
- 包含文章的**核心价值承诺**（读完能获得什么）
- 用一句话制造"想点开看"的冲动
- 不要简单重复标题内容
- 可以补充标题未提到的信息点
- 适当使用数字增强具体感

**示例：**
- 标题：「为什么你学了那么多课，还是没有变化？」
- 摘要：「对比了 200+ 知识付费用户的学习数据，发现真正有效的不是选什么课，而是一个被 90% 的人忽略的学习习惯。附实操清单。」

### 4.5 风格参数化

在创作前确认以下参数（如用户未指定，使用合理默认值并告知用户）：

**Tone of Voice 矩阵：**

| 风格 | 特征 | 句式特点 | 适用场景 |
|------|------|----------|----------|
| 正式严谨 | 客观、学术感、数据驱动 | 长句、被动语态、专业术语 | 行业报告、政策解读 |
| 轻松专业 | 专业但不端着，有人味 | 短句为主、偶尔口语、比喻 | 干货教程、产品介绍（**默认**）|
| 亲和幽默 | 像朋友聊天，有梗 | 反问、自嘲、网络用语 | 生活方式、娱乐向 |
| 犀利锐评 | 观点鲜明，不怕得罪人 | 短促有力、反转、金句 | 观点评论、行业批评 |
| 温暖治愈 | 共情、鼓励、有温度 | 第二人称、排比、意象 | 人物故事、情感向 |

**目标读者画像：**
- 年龄段 + 性别倾向
- 职业/行业
- 核心关注点和痛点
- 阅读习惯（碎片化/深度）
- 知识水平（小白/进阶/专业）

**参考风格对标：**
- 如用户提供示例文章，使用 WebFetch 抓取并分析其：用词习惯、句式长度、段落节奏、情感基调、互动方式
- 提取风格 DNA 并在创作中模仿

## 五、排版规范

排版质量直接决定读者的阅读体验和完读率。微信公众号 90%+ 的阅读发生在手机端，所有排版决策以 375px 宽度的移动端屏幕为基准。

### 5.1 基础排版参数

| 参数 | 推荐值 | 允许范围 | 说明 |
|------|--------|----------|------|
| 正文字号 | 15px | 14-16px | 15px 是手机端最佳可读性 |
| 行距 | 1.8 | 1.75-2.0 | 低于1.75 显得拥挤，高于2.0 显得稀疏 |
| 字色 | #3f3f3f | #333-#555 | 避免纯黑 #000（对比过强伤眼） |
| 段间距 | 20px | 15-25px | 段间留白比换行符更舒适 |
| 对齐方式 | justify | - | 两端对齐，中文排版标准 |
| 首行缩进 | 不缩进 | - | 手机端不缩进是主流做法 |
| 左右边距 | 内容区两侧各留 8-16px | - | 微信编辑器自带边距，一般不需额外设置 |

### 5.2 文字层级体系

一篇文章需要清晰的视觉层级，让读者能快速扫描和定位：

```
大标题（h2）：18px bold，#1a1a1a，左侧4px色条，上间距30px
小标题（h3）：16px bold，#2a2a2a，上间距25px
正文：15px，#3f3f3f，行距1.8
辅助说明：13px，#999，用于图片说明、来源标注
强调文字：15px bold 或 主色调着色
引用文字：14px，#666，灰色背景块
```

### 5.3 段落与节奏控制

**段落长度：**
- 每段 **2-4 句话**，单段不超过 **100 字**
- 超过 4 句必须拆段
- 段间用 `margin-bottom: 20px` 分隔（不用连续 `<br>`）

**视觉节奏（每屏变化原则）：**

一屏大约 400-500 字，每一屏都应该有视觉变化，避免"文字墙"：

```
第1屏：标题 + 引导语 + 开头Hook
第2屏：正文段落 + 小标题
第3屏：正文段落 + 【图片】
第4屏：正文段落 + 【引用块/色块】
第5屏：正文段落 + 【图片】
第6屏：正文段落 + 【数据可视化/表格】
...
末屏：总结 + CTA
```

**视觉元素穿插频率：**
- 图片：每 3-4 段一张
- 引用块/色块：每 5-6 段一个
- 小标题：每 300-500 字一个
- 分割线：用于章节切换，不要过多（整篇最多 3-4 条）

### 5.4 配色方案

提供 4 套预设配色，用户可自定义或提供品牌色：

**方案一：商务蓝（默认，适合大多数场景）**
```
主色调：#1e88e5    强调色：#ff6b35
引用背景：#f7f7f8   色块背景：#e3f2fd
分割线：#eaeaea    小标题色条：#1e88e5
```

**方案二：科技紫（适合技术、创新类）**
```
主色调：#7c4dff    强调色：#ff5252
引用背景：#f3f0ff   色块背景：#ede7f6
分割线：#e0e0e0    小标题色条：#7c4dff
```

**方案三：自然绿（适合健康、教育类）**
```
主色调：#43a047    强调色：#ff8f00
引用背景：#f1f8f1   色块背景：#e8f5e9
分割线：#e0e0e0    小标题色条：#43a047
```

**方案四：中国红（适合传统文化、节日类）**
```

引用背景：#fef8f8   色块背景：#ffebee
分割线：#eaeaea    小标题色条：#c62828
```

如用户有品牌色，以品牌色为主色调，自动推导其他色值（互补色做强调色，主色10%透明度做背景色）。

### 5.5 图片规范

**图片类型与尺寸：**
| 位置 | 推荐宽度 | 宽高比 | 格式 |
|------|----------|--------|------|
| 封面图 | 900px+ | 2.35:1（横图优先） | JPG |
| 正文配图 | 750-1080px | 不限，16:9 / 4:3 / 1:1 | JPG/PNG |
| 信息图/流程图 | 750px | 按内容自适应 | PNG |
| GIF动图 | 750px | 不限 | GIF（<2MB） |

**图片样式：**
```html
<!-- 标准图片：全宽、圆角、居中说明 -->
<section style="text-align: center; margin: 20px 0;">
  <img src="微信域名URL" style="width: 100%; height: auto; border-radius: 6px;" />
  <p style="font-size: 12px; color: #999; margin-top: 8px; line-height: 1.5;">▲ 图片说明文字（来源：XXX）</p>
</section>
```

**图片占位标记格式：**
```html
<!-- IMAGE: {描述} | 建议类型：{照片/插画/截图/数据图} | 建议风格：{写实/扁平/手绘} -->
```

### 5.6 特殊排版组件

在 `references/wechat_html_compat.md` 中提供以下常用组件的 HTML 模板：

- **重点提示框**（浅色背景 + 左侧色条）
- **步骤编号列表**（彩色圆形编号 + 标题 + 说明）
- **对比表格**（双栏对比，交替行背景）
- **数据高亮卡**（大字号数字 + 小字说明）
- **名言引用**（大引号装饰 + 来源标注）
- **分割线变体**（虚线、圆点、渐变线、图标分割）
- **文末作者卡**（头像 + 简介 + 关注引导）

## 六、图片获取与插入（必做环节）

公众号文章没有图片就等于没有排版。图片不是可选项，是必须项。每篇文章至少需要 1 张封面图 + 2-4 张正文配图。

### 6.1 图片来源优先级

按以下优先级获取图片，优先使用高优先级来源：

**优先级 1：用户提供的素材**
- 用户直接提供的本地文件路径
- 用户提供的图片 URL
- 用户指定的素材目录

**优先级 2：搜索免费可商用图片**
- 使用 WebSearch 搜索 Unsplash / Pexels / Pixabay 上的相关图片
- 使用 WebFetch 下载选定的图片到本地
- 确保图片许可证允许商用（CC0 或类似协议）

**优先级 3：用户自有素材库**
- 检查项目目录下的 `images/` 或 `assets/` 目录
- 使用 Glob 扫描可用的图片文件
- 根据文件名和上下文匹配合适的图片

**优先级 4：明确告诉用户需要什么图片**
- 列出图片清单，说明每张图的用途、建议尺寸和风格
- 给出具体的搜索建议词，方便用户自行寻找

### 6.2 图片处理流程

```
获取图片 -> 检查格式/尺寸 -> 上传到微信服务器 -> 获取微信域名URL -> 替换HTML中的占位符
```

具体步骤：

1. **获取图片**：按 6.1 的优先级获取图片文件
2. **检查格式/尺寸**：确认格式为 JPG/PNG/GIF，大小不超过 10MB，分辨率满足要求
3. **上传到微信服务器**：使用 `python3 scripts/wechat_api.py upload_image <image_path>` 上传
4. **获取微信域名 URL**：上传成功后获取 `mmbiz.qpic.cn` 域名的图片地址
5. **替换 HTML 中的占位符**：将正文中的 `<!-- IMAGE: ... -->` 占位符替换为实际的 `<img>` 标签

### 6.3 图片选择原则

- **封面图**：高饱和度、简洁有冲击力，在小缩略图下也能看清主体
- **正文配图**：与内容直接相关，不放纯装饰性图片，每张图都要为内容服务
- **数据类内容**：优先用截图或数据可视化图表，比文字描述更有说服力
- **禁止使用**：AI 生成的插画风格图片（一眼假）、低分辨率图、带水印图、与内容无关的装饰图

### 6.4 图片与内容的强关联策略

图片不是"配"文章的，是文章的一部分。每张图都必须让读者觉得"这张图就是为这段话准备的"。

**搜索策略——按内容实体搜索，而非按主题搜索：**

错误做法：文章讲AI行业 → 搜索 "AI technology" → 得到一堆泛泛的科技感图片
正确做法：根据每段具体内容，提取关键实体词来搜索

| 文章内容 | 错误搜索词 | 正确搜索词 |
|----------|-----------|-----------|
| 提到OpenAI和Anthropic竞争 | "AI competition" | "OpenAI logo"、"Anthropic Claude" |
| 提到MCP协议安装量9700万 | "protocol network" | "Model Context Protocol"、"API connection diagram" |
| 提到Block裁员股价涨 | "layoff technology" | "Block company Jack Dorsey"、"stock market chart 2026" |
| 提到AI Native一人公司 | "startup office" | "solo entrepreneur laptop"、"one person startup working" |

**不同内容类型的图片匹配规则：**

| 内容类型 | 优先图片类型 | 搜索方式 |
|----------|------------|----------|
| 提到具体公司/产品 | 该公司/产品的官方截图、logo、产品界面 | WebSearch 搜公司名+screenshot/logo |
| 提到数据/报告 | 原报告中的图表截图 | WebFetch 抓取报告原文，截取关键图表 |
| 提到具体人物 | 该人物的公开照片 | WebSearch 搜人名+photo（注意版权） |
| 提到行业趋势 | 相关行业的真实场景照片 | 搜具体场景而非抽象概念 |
| 提到技术概念 | 技术架构图、流程图、官方文档截图 | 搜技术名词+architecture/diagram |

**封面图的特殊策略：**
- 封面图要能"一图说清文章主题"
- 如果文章讲竞争格局，用多个公司logo拼图比抽象科技背景好
- 如果文章讲某个事件，用事件相关的真实照片
- 如果找不到完美匹配的，优先用文章中最有冲击力的数据做成文字海报风格的封面

### 6.5 无法获取图片时的处理

不要留着占位符就交付给用户。如果无法自动获取图片，必须：

1. 列出完整的图片需求清单，包括：
   - 每张图的用途（封面 / 正文第X处配图）
   - 建议的图片内容描述
   - 建议的尺寸和比例
   - 推荐的搜索关键词
2. 明确告诉用户："文章正文已完成，但还需要 X 张图片才能发布，以下是图片清单..."
3. 等用户提供图片后再完成上传和替换

## 七、输出格式

### 7.1 微信兼容 HTML

微信公众号编辑器支持的是 HTML 子集，有严格限制。详细兼容性说明见 `references/wechat_html_compat.md`。

**核心限制（务必遵守，否则排版会错乱）：**
- **所有样式必须 inline** — `class`、`id` 属性会被过滤，`<style>` 标签无效
- **不支持的标签** — `<script>`、`<iframe>`、`<form>`、`<input>`、`<video>`（需用微信自有组件）
- **不支持的 CSS** — `position: fixed/absolute`、`float`（不推荐）、`animation`、`@media`、外部字体
- **图片 src** — 必须为 `mmbiz.qpic.cn` 域名（发布前通过 API 上传获取）
- **超链接** — 仅支持公众号文章互链和白名单域名，普通外链点击无效

**HTML 输出结构模板：**

```html
<section style="padding: 0; margin: 0;">

  <!-- 引导语（可选） -->
  <p style="font-size: 13px; color: #999; text-align: center; margin-bottom: 25px; line-height: 1.6;">
    点击上方蓝字关注我们
  </p>

  <!-- 正文内容 -->
  <section style="font-size: 15px; color: #3f3f3f; line-height: 1.8; text-align: justify;">

    <!-- 段落 -->
    <p style="margin-bottom: 20px;">正文内容...</p>

    <!-- 小标题 -->
    <h2 style="font-size: 18px; font-weight: bold; color: #1a1a1a;
               border-left: 4px solid #1e88e5; padding-left: 12px;
               margin: 30px 0 15px 0;">章节标题</h2>

    <!-- 图片 -->
    <section style="text-align: center; margin: 20px 0;">
      <img src="mmbiz.qpic.cn域名URL" style="width: 100%; border-radius: 6px;" />
      <p style="font-size: 12px; color: #999; margin-top: 8px;">▲ 说明</p>
    </section>

    <!-- 引用块 -->
    <blockquote style="border-left: 3px solid #1e88e5; padding: 12px 16px;
                       background: #f7f7f8; margin: 15px 0; border-radius: 0 4px 4px 0;">
      <p style="font-size: 14px; color: #666; margin: 0; line-height: 1.8;">引用内容</p>
    </blockquote>

    <!-- 强调色块 -->
    <section style="background: #e3f2fd; padding: 16px 20px; border-radius: 6px; margin: 15px 0;">
      <p style="font-size: 15px; color: #1565c0; margin: 0; line-height: 1.8;">
        <strong>重点：</strong>强调内容
      </p>
    </section>

  </section>

  <!-- 文末CTA -->
  <section style="text-align: center; margin-top: 30px; padding-top: 20px;
                  border-top: 1px dashed #eaeaea;">
    <p style="font-size: 14px; color: #999;">CTA引导文字</p>
  </section>

</section>
```

**输出后自检清单：**
1. 所有 `<img>` 的 src 是否为微信域名或占位标记？
2. 是否所有样式都是 inline 的？没有 class/id？
3. 是否有外链 `<a>` 指向非白名单域名？
4. 段落间距是否一致？
5. 字号、颜色是否符合配色方案？
6. 总字数是否在 1500-3000 范围内？

### 7.2 Markdown 输出（可选，用于第三方编辑器）

用户可能需要将内容导入秀米、135编辑器、壹伴等第三方工具。在 HTML 输出后询问是否需要 Markdown 版本。

**Markdown 格式要求：**
- 使用标准 CommonMark 语法
- 图片占位使用 `![描述](IMAGE_PLACEHOLDER)` 格式
- 保留标题层级（`##` 和 `###`）
- 保留加粗、引用、列表等格式
- 去掉所有 HTML 特有的样式信息
- 在文件开头添加 YAML front matter：

```markdown
---
title: "文章标题"
author: "作者"
date: "YYYY-MM-DD"
digest: "摘要内容"
tags: [标签1, 标签2]
---
```

### 7.3 纯文本输出（可选，用于社群转发）

部分用户需要将核心内容转发到社群/朋友圈，提供精简版纯文本：
- 去掉所有格式标记
- 保留核心观点和金句
- 控制在 300 字以内
- 适合复制粘贴到微信聊天

## 八、发布工作流

这是核心流程，每一步都有对应的 API 调用。脚本位于 `scripts/wechat_api.py`。

### Step 1：创作内容

按照第三、四、五、七节的规范，生成：
- HTML 正文
- 标题（用户已确认的）
- 摘要（120字以内）

### Step 2：上传素材

**正文内图片**：
```bash
python3 scripts/wechat_api.py upload_image <image_path>
# POST https://api.weixin.qq.com/cgi-bin/media/uploadimg
# 返回微信域名 URL，替换正文中的图片 src
```

**封面图**：
```bash
python3 scripts/wechat_api.py upload_material <image_path> --type image
# POST https://api.weixin.qq.com/cgi-bin/material/add_material
# 返回 media_id 和 url
```

如果用户没有准备图片，提示用户提供，或建议使用已有素材库中的图片。

### Step 3：添加草稿

```bash
python3 scripts/wechat_api.py add_draft \
  --title "文章标题" \
  --content "<html内容>" \
  --digest "文章摘要" \
  --thumb_media_id "封面图media_id" \
  --author "作者名" \
  --need_open_comment 1 \
  --only_fans_can_comment 0
# POST https://api.weixin.qq.com/cgi-bin/draft/add
# 返回草稿 media_id
```

### Step 4：预览

```bash
python3 scripts/wechat_api.py preview \
  --media_id "草稿media_id" \
  --towxname "用户微信号"
# POST https://api.weixin.qq.com/cgi-bin/message/mass/preview
# 发送预览到用户微信
```

**⚠️ 关键暂停点**：发送预览后，必须停下来告诉用户：

> 预览已发送到你的微信，请在手机上查看实际效果。确认无误后请回复"确认发布"或"可以发"，如需修改请告诉我具体问题。

**绝不**在用户确认前执行下一步。

### Step 5：发布

用户明确确认后（识别关键词：确认发布、可以发、发吧、没问题、OK等）：

```bash
python3 scripts/wechat_api.py publish --media_id "草稿media_id"
```

## 九、SEO 与传播优化

微信公众号的 SEO 有两个维度：**微信搜一搜排名** 和 **社交传播裂变**。两者策略不同，需要兼顾。

### 9.1 微信搜一搜 SEO

微信搜一搜已成为重要的公域流量入口，文章的搜索排名由以下因素决定：

**关键词布局策略：**

| 位置 | 权重 | 策略 |
|------|------|------|
| 标题 | ★★★★★ | 核心关键词放在前 15 字内 |
| 公众号名称 | ★★★★ | 账号命名与内容领域一致 |
| 摘要/digest | ★★★★ | 包含 1-2 个关键词 |
| 首段（前 100 字） | ★★★★ | 自然融入核心关键词和长尾词 |
| 小标题（h2/h3） | ★★★ | 每个小标题包含关键词变体 |
| 正文 | ★★ | 关键词密度 2-3%，使用同义词/近义词分散 |
| 图片 alt/说明 | ★ | 图片说明文字包含关键词 |

**关键词挖掘方法（在信息搜集阶段完成）：**
1. 用 WebSearch 搜索目标关键词，查看搜一搜联想词
2. 分析排名靠前的公众号文章标题用词
3. 记录长尾关键词（如：不只搜"AI"，还要覆盖"AI工具推荐""AI替代人工""AI学习路线"）
4. 整理关键词表：1 个核心词 + 3-5 个长尾词 + 5-8 个相关词

**正文中的关键词使用：**
- 核心关键词出现 3-5 次（标题、首段、小标题、末段）
- 长尾关键词各出现 1-2 次
- 使用同义词替换避免堆砌（如"人工智能""AI""智能技术"交替）
- 关键词出现位置要自然，不影响阅读体验

### 9.2 社交传播优化

公众号文章的传播主要靠：**朋友圈分享** + **群聊转发** + **在看/点赞推荐**。

**提升分享率的内容要素：**

| 要素 | 说明 | 实现方式 |
|------|------|----------|
| 社交货币 | 让读者分享后显得"有品位/有见识" | 独到观点、前沿信息、深度分析 |
| 实用价值 | "这个对你有用，转给你看看" | 清单、教程、工具推荐 |
| 情感共鸣 | 触发"我也是这样"的共鸣 | 痛点描述、真实故事、温暖治愈 |
| 争议话题 | 激发讨论欲，"你觉得呢？" | 观点评论、对比分析、行业争议 |

**分享卡片优化（标题+摘要+封面）：**

朋友圈和群聊中只显示标题+摘要+封面缩略图，这三样决定了 70% 的打开率：
- **标题**：见 3.1 节标题打磨规则
- **摘要**：见 3.4 节，摘要要补充标题的信息缺口
- **封面**：高饱和度、大字报式优先（手机上缩略图很小，复杂图片看不清）

### 9.3 发布时间策略

| 读者画像 | 黄金时段 | 次优时段 | 原因 |
|----------|----------|----------|------|
| 职场白领 | 7:30-8:30（通勤） | 12:00-13:00 / 21:00-22:00 | 通勤刷手机、午休、睡前 |
| 学生群体 | 21:30-23:00（晚自习后） | 12:00-13:30 | 午休和睡前 |
| 宝妈群体 | 9:30-11:00（孩子上学后） | 14:00-15:30 | 上午空闲、午后空闲 |
| 自由职业 | 10:00-11:00 | 20:00-21:00 | 上午工作状态好、晚间放松 |
| 企业管理者 | 6:30-7:30（早起型） | 22:00-23:00 | 早起学习、深夜思考 |
| 通用 | **20:00-22:00** | 12:00-13:00 | 全民阅读高峰 |

**额外时间策略：**
- 周二到周四的完读率通常高于周一和周末
- 节假日阅读量通常下降 20-30%，除非内容与节日相关
- 热点事件发生后 2-4 小时内发布蹭热点文章效果最佳

### 9.4 转发引导体系

**文末引导话术（根据内容类型选择）：**

| 内容类型 | 引导话术 |
|----------|----------|
| 干货教程 | 「收藏 = 吃灰，转发 = 真学。把这篇发给你身边也需要的朋友吧。」 |
| 观点评论 | 「你怎么看？评论区聊聊。点赞最高的留言我会认真回复。」 |
| 行业分析 | 「觉得有启发？点个"在看"，推荐给更多同行。」 |
| 人物故事 | 「每个人都有自己的故事。如果你也有共鸣，转发给懂你的人。」 |
| 产品介绍 | 「觉得不错？推荐给你觉得需要的朋友，好东西值得分享。」 |

**评论区运营建议（发布后）：**
- 预埋 1-2 条引导性评论（如"看完的小伙伴，说说你们的经历？"）
- 对高质量留言快速回复（提升互动数据）
- 置顶有价值的用户留言（增加社区感）

### 9.5 系列文章与矩阵策略

如果用户有持续运营需求，建议：
- **系列化**：将一个大主题拆为 3-5 篇系列文章，每篇末尾互相链接
- **首篇引流**：首篇做高传播性内容（清单/盘点），后续做深度内容
- **交叉推荐**：在文中自然嵌入历史文章链接（"我之前写过 → {链接}"）
- **固定栏目**：建议用户固定发布栏目（如"每周一XX"），培养读者预期

## 十、合规与风控

发布前自动执行以下检查：

### 敏感词检测

运行检查脚本：
```bash
python3 scripts/compliance_check.py --content "<html内容>"
```

检查项：
- **广告法极限词**：最好、最佳、第一、顶级、绝对、100%、永久 等
- **违禁词**：涉政敏感词、低俗词汇
- **医疗/金融声明**：如涉及健康或理财内容，提醒添加免责声明

如检测到问题，列出具体位置和替换建议，让用户确认修改。

### 原创声明

询问用户是否标记为原创文章（`need_open_comment` 和原创标记）。原创文章享受更高推荐权重，但必须确保内容确实为原创。

### 发布前二次确认

即使用户已在预览阶段确认，在实际调用发布API前再次确认：

> 即将发布文章「{标题}」到你的公众号。这是正式发布，发布后所有关注者可见。确认发布？

## 十一、错误处理

### access_token 过期

所有 API 调用失败时检查 errcode：
- `40001`/`42001`：token 过期，自动刷新后重试（最多重试 2 次）
- 刷新逻辑内置于 `scripts/wechat_api.py`

### 素材上传失败

- 检查文件格式（支持 jpg/png/gif，大小限制 10MB）
- 检查网络连接
- 重试 1 次，仍失败则提示用户并给出具体错误信息

### API 限流

- 每日接口调用有次数限制（通常 5000 次/天）
- 遇到限流错误（errcode 45009），提示用户稍后再试
- 建议避免频繁测试，善用预览功能

### 通用错误处理

所有 API 返回的 errcode 非 0 时，脚本会输出：
```
错误码：{errcode}
错误信息：{errmsg}
建议操作：{具体建议}
```

常见错误码对照见 `references/error_codes.md`。
