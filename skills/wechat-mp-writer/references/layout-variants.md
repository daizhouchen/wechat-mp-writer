# 排版骨架库（v2 防审美疲劳核心）

## 0. 为什么要这份文档

**问题**：v1 SKILL.md §5 给的是一套"通用排版规范"——但每篇都用一套规范，结果就是连续读三篇这个号的文章，读者会觉得"全是 AI 写的、长得都一样"。

**解决方案**：把"通用排版"拆成 **7 种骨架 + 视觉装饰库 + 防重复机制**。每篇写之前先选骨架（落 `meta.layout_variant.id`），写完再扫"近 5 篇是否结构重复"。

**对标参考**：
- 新智元：每篇结构差异明显，有时数据图开篇、有时人物访谈摘录开篇、有时直接 GitHub README 截图开篇
- 量子位：擅长时间线 + 对比表混合
- 36氪深度：故事开篇 + 多小节平铺
- 财新：访谈摘录 + 时间线，少装饰

**字数约束（重申）**：
- short：1500-2500 字（占 60%，日更主力）
- medium：2500-4000 字（占 30%，深度选题）
- long：4000-6000 字（占 10%，专题/年终盘点）
- **绝对上限 6000 字**——distiller 风格的 8000+ 字硬上限不要搬

## 1. 七种骨架（layout_variant.id enum）

### 骨架 1：`data_lead`（数据先行式）

**适用**：评测对比、行业数据稿、市场份额变化、价格对比。
**节奏**：
```
[首屏] 一张大图表 / 一组对比数字  (200 字内)
   ↓
[第二屏] 拆解这组数据怎么得到的  (500-800 字)
   ↓
[中段] 数据背后的机制 / 玩家动作  (1000-1500 字)
   ↓
[末段] 数据指向的趋势判断 + CTA  (200-400 字)
```

**典型 H2 模式**：
- 「先看三个数字」
- 「这三个数字是怎么来的」
- 「数字背后藏着什么」
- 「下半年会怎么变」

**视觉特点**：开头大量数字 + 图表，中段叙述化，末段判断收尾。

---

### 骨架 2：`story_open`（故事开篇式）

**适用**：人物报道、事件还原、公司案例、产品上线纪实。
**节奏**：
```
[首屏] 一个具体场景 / 一段对话 / 一个时刻  (200-300 字)
   ↓
[第二屏] 这件事的背景 / 主角是谁  (400-600 字)
   ↓
[中段] 时间线推进 / 决策转折 / 关键冲突  (1500-2500 字)
   ↓
[末段] 这件事意味着什么  (300-500 字)
```

**典型 H2 模式**：
- 「1 月 20 日晚上 9 点」（具体时间锚点）
- 「梁文锋的另一面」（人物切入）
- 「Sam Altman 的反应」（反方视角）
- 「这次发布改变了什么」

**视觉特点**：开头一张人物 / 场景照，中段配产品截图 / 时间线图。

---

### 骨架 3：`qa_list`（问答列表式）

**适用**：产品深扒、技术原理科普、政策解读、新名词解释。
**节奏**：
```
[首屏] 引出问题 + 列出 N 个问题清单  (200-300 字)
   ↓
[Q1] 「X 到底是什么？」 + 答  (300-500 字)
[Q2] 「为什么现在火？」 + 答  (300-500 字)
[Q3] 「和 Y 有什么区别？」 + 答  (300-500 字)
[Q4-7] 同上
   ↓
[末段] 还没解决的问题 + CTA  (200-400 字)
```

**典型 H2 模式**：
- Q&A 编号格式：「Q1. R1 到底是什么？」「Q2. 它和 o1 比强在哪？」
- 或问句格式：「R1 凭什么干掉 o1？」「为什么现在才出现？」

**视觉特点**：每个 Q 用突出色块，A 用正文样式。

---

### 骨架 4：`timeline`（时间线式）

**适用**：公司发展史、技术演进、行业大事记、年终盘点、危机回溯。
**节奏**：
```
[首屏] 当下的结局 / 现状  (200-300 字)
   ↓
[时间点 1] 「2017 年」 + 当时发生了什么  (300-500 字)
[时间点 2] 「2020 年」 + 转折点  (300-500 字)
[时间点 3] 「2023 年」 + 加速  (300-500 字)
[时间点 N] ...
   ↓
[末段] 时间线告诉我们什么  (300-500 字)
```

**典型 H2 模式**：直接用年份 / 季度 / 月份作 H2（「2017：起点」「2020：转折」）

**视觉特点**：开头一张时间轴 SVG，每节配该年份的标志性图。

---

### 骨架 5：`comparison_table`（对比表式）

**适用**：产品横评、模型对比、方案选型、竞品分析。
**节奏**：
```
[首屏] 抛出对比需求 + 总结一句  (200-300 字)
   ↓
[核心对比表] 一张大表，多维度  (占首屏第二屏)
   ↓
[维度 1 深拆] 表里某一列展开  (400-600 字)
[维度 2 深拆] 表里另一列展开  (400-600 字)
[维度 3 深拆] ...
   ↓
[末段] 给不同需求的建议（场景化推荐）  (300-500 字)
```

**典型 H2 模式**：
- 「先看总表」
- 「价格对比拆开看」
- 「能力对比拆开看」
- 「该选谁——按场景看」

**视觉特点**：开头一张大对比表是文章核心视觉锚点，中段配每个维度的图。

---

### 骨架 6：`list_breakdown`（拆解清单式）

**适用**：经验总结、复盘、避坑指南、上手教程、Top X 榜单。
**节奏**：
```
[首屏] 定主题 + 预告 N 个要点  (200-300 字)
   ↓
[要点 1] 标题 + 内容 + 例子  (300-500 字)
[要点 2] 同上
[要点 3-N] 同上（一般 3-7 个，超过 7 个建议拆系列）
   ↓
[末段] 综合 + 进阶资源 / 下一步  (200-400 字)
```

**典型 H2 模式**：
- 数字开头：「01. 第一个误区」「02. 第二个误区」
- 中文序号：「其一」「其二」「其三」
- 或直接用核心词：「先搞清楚定位」「再看用户画像」

**视觉特点**：每个要点用统一的小标题装饰（数字 + 短线），简洁不花哨。

---

### 骨架 7：`quote_montage`（访谈摘录 / 引语集）

**适用**：人物访谈整理、行业大佬观点合集、内部信解读、X / 微博体合辑。
**节奏**：
```
[首屏] 引出 N 位说话人 + 共同话题  (200-300 字)
   ↓
[引语块 1] 大字引语 + 谁说的 + 时间 + 短解读  (300-400 字)
[引语块 2] 同上
[引语块 3-N] 同上
   ↓
[末段] 把这些观点拼起来看  (300-500 字)
```

**典型 H2 模式**：
- 直接用说话人名字：「Sam Altman：……」「梁文锋：……」
- 或主题切片：「关于训练成本」「关于开源」「关于监管」

**视觉特点**：引语块是核心视觉，大字 + 引号 + 朱砂色，每块配说话人头像。

## 2. 防重复机制

### 历史读取

每篇文章生成时，SKILL.md §4.2 必须先读 `~/.wechat-history.json`（或本地 `./articles/_history.json`）：

```json
{
  "recent": [
    {"slug": "deepseek-r1-reasoning", "variant_id": "story_open", "created_at": "2026-05-10"},
    {"slug": "anthropic-funding", "variant_id": "data_lead", "created_at": "2026-05-08"},
    {"slug": "ai-regulation-eu", "variant_id": "qa_list", "created_at": "2026-05-05"},
    {"slug": "...", "variant_id": "story_open", "created_at": "2026-05-03"},
    {"slug": "...", "variant_id": "timeline", "created_at": "2026-05-01"}
  ]
}
```

落到 article.json 的 `meta.recent_layouts`。

### 选择规则

| 情况 | 规则 |
|------|------|
| 近 3 篇有同一 variant | 禁用该 variant |
| 近 5 篇有 ≥ 2 次同一 variant | 严重警告，强烈建议换 |
| 近 5 篇都不同 variant | 自由选 |
| 用户强制指定 variant | 用户指定优先（覆盖防重复规则），但提示"近期已用过" |

### quality_check.py 强制检查

`quality_check.py` 加一项硬指标：
- `layout_variety_check`：扫近 5 篇的 variant_id 集合，重复率 ≤ 40% → pass；> 40% → warn

## 3. 视觉装饰库（每篇换装饰）

骨架定了"骨头"，下面这套装饰库定"皮肉"——每篇换一套，避免视觉重复。

### H2 装饰风格（h2_decoration）

| id | 视觉 | 适用骨架 | HTML 模板 |
|---|---|---|---|
| `ink_underline` | 朱砂细线下划 | data_lead / story_open | `<h2 style="border-bottom:2px solid #8b3a3a; padding-bottom:8px; ...">` |
| `block_left_bar` | 左侧色块 | qa_list / list_breakdown | `<h2 style="border-left:5px solid #8b3a3a; padding-left:12px; ...">` |
| `number_prefix` | 数字编号前缀 | list_breakdown / timeline | `<h2><span style="color:#8b3a3a; font-size:120%;">01</span> 标题</h2>` |
| `chinese_serial` | 中文序号 | list_breakdown | `<h2><span style="font-family: serif;">其一</span>　标题</h2>` |
| `full_block` | 整行底色 | comparison_table / quote_montage | `<h2 style="background:#f5f0e8; padding:10px 16px; ...">` |
| `dash_decoration` | 短线装饰 | story_open / quote_montage | `<h2>—— 标题 ——</h2>` |
| `large_number` | 大字编号在侧 | timeline | 自定义 div + 大数字 |

### 强调色（accent_color）

不要每篇都用朱砂 `#8b3a3a`。按主题切换：

| 主题类型 | 推荐 accent | 备色 |
|---|---|---|
| 科技 / AI | `#1e4d8b` 深青 | `#8b3a3a` 朱砂 |
| 财经 / 商业 | `#7a5500` 暖橙 | `#1e4d8b` 深青 |
| 人物 / 故事 | `#8b3a3a` 朱砂 | `#5c4a3a` 大地色 |
| 监管 / 政策 | `#3a3a3a` 墨黑 | `#7a5500` 暖橙 |
| 评测 / 横评 | `#1e4d8b` 深青 | `#8b3a3a` 朱砂 |
| 复盘 / 教程 | `#5c4a3a` 大地色 | `#7a7a7a` 中灰 |

**强制约束**：accent_color 必须和"近 5 篇用过的色"不同。落到 `meta.layout_variant.accent_color`。

### 图文节奏（image_rhythm）

| id | 节奏 | 适用骨架 |
|---|---|---|
| `front_heavy` | 前 1/3 内容集中放 60% 图 | data_lead / story_open / comparison_table |
| `even` | 全文均匀分布 | qa_list / list_breakdown |
| `tail_heavy` | 后 1/3 内容多图 | timeline / quote_montage |

落到 `meta.layout_variant.image_rhythm`，`images_plan[]` 的 `section_id` 分布要符合本字段。

### 段落装饰（轮换使用）

每篇随机选 1-2 种段落装饰（不要全用）：

- **引述块**：左侧朱砂竖线 + 微缩进 + 浅灰底
- **数字强调段**：大字数字 + 一句话解释
- **对比卡片**：左右两栏小卡片
- **批注段**：右侧灰字小段如同手写批注
- **小标签**：句末 `<span style="...">据 X 推算</span>` 小标签

**禁忌**：一篇文章里同一种段落装饰用 ≥ 3 次（重复感）。

## 4. 骨架选择决策树

不要每次都问"用哪种骨架"——SKILL.md §4.2 按下面决策树自动选：

```
1. 主题里有 ≥ 3 个具体数字？ → data_lead 或 comparison_table
   ├─ 是横评/对比 → comparison_table
   └─ 是单主题数据深扒 → data_lead

2. 主题是"一个具体的人/事件/产品上线"？ → story_open

3. 主题是"科普 / 解释新名词"？ → qa_list

4. 主题跨度 ≥ 6 个月或多个时间节点？ → timeline

5. 主题是"经验总结 / 教程 / 避坑 / 榜单"？ → list_breakdown

6. 主题是"多人观点合集 / 访谈整理"？ → quote_montage

7. 都不像？ → 选 audience_fit 评分最高的角度对应的最自然骨架
```

**强制覆写规则**：如果决策树选的 variant 和"近 3 篇"有重复，必须换次优。例如近 3 篇都用过 story_open，本篇即使是人物报道也要换 qa_list 或 timeline。

## 5. 字数 budget 与 tier 对应

| tier | target | min | max | 适用骨架（推荐） |
|------|--------|-----|-----|------------------|
| short | 2000 | 1500 | 2500 | qa_list / list_breakdown |
| medium | 3000 | 2500 | 4000 | data_lead / story_open / comparison_table |
| long | 5000 | 4000 | 6000 | timeline / quote_montage / 多骨架混用 |

`quality_check.py` 加硬阈值：
- `word_count < min` → warn（单薄）
- `word_count > max` → warn（超出 tier）
- `word_count > 6000` → fail（绝对上限）

**默认策略**：用户没说时，tier 按 .wechat-profile.md 的 `default_tier` 取（首次画像时引导填）。

## 7. 自检题

写完一篇前问自己：

1. 本篇 `layout_variant.id` 是不是和近 3 篇都不同？
2. 本篇 `accent_color` 是不是和近 3 篇都不同？
3. 字数有没有在 `word_budget.min`-`word_budget.max` 之间？
4. 有没有超过 6000 字硬上限？
5. H2 装饰风格是不是和骨架匹配（不要 timeline 骨架用 chinese_serial 装饰这种错配）？
6. `image_rhythm` 和实际 `images_plan` 分布是不是一致？
7. 段落装饰有没有任何一种用 ≥ 3 次？

| 骨架 | 字数 sweet spot | 配图密度 | 难度 | 重复风险 |
|------|---|---|---|---|
| data_lead | 2500-3500 | 中（前重） | 中 | 中（数据稿太多） |
| story_open | 3000-5000 | 高（前重 + 中段） | 高 | 中 |
| qa_list | 1800-2800 | 低（每 Q 一图就够） | 低 | 高（最容易重复） |
| timeline | 3000-5000 | 中（每节点一图） | 中 | 低（用得少） |
| comparison_table | 2500-3500 | 中（核心是表） | 中 | 中 |
